# PostgreSQL Migration Plan

This is the working migration direction for restarting the old Douban/WeRead project. It is based on the real local audit generated by `tools/audit_legacy_data.py`.

## Current Assets

### Keep as source of truth

- MySQL `douban.weixin`: 173,515 WeRead rows. This is the best legacy source for WeRead catalog, shelf flags, paid flags, archive fields, copyright info, and old import timestamps.
- MySQL `douban.weixin_douban`: 19,117 manual WeRead-Douban links. This is high-value human curation and must be migrated intact.
- MySQL `douban.weixin_booklist` and `douban.weixin_booklist_book`: old booklist metadata and 56,647 booklist links.
- MySQL `douban.weixin_pay_history`: 25,032 historical payment rows. New WeRead skill does not appear to expose payment history, so this becomes historical-only data.
- MySQL `douban.author`, `douban.meta_book`, `douban.weixin_meta_book`: small but valuable manual author/work normalization.
- Mongo BSON `backup/mongo.out/local/book.bson`: 490,229 full Douban API documents. This is richer than SQL `books` and richer than Elasticsearch.

### Use only as derived or fallback data

- MySQL `douban.books`: 154,062 SQL-projected Douban rows. Use for legacy UI parity, status flags, and old crawler fields, but prefer Mongo BSON for bibliographic details.
- MySQL `douban.douban_props`: 323,708 key/value facts for 38,360 Douban books. Use as supplemental facts, especially translator, publisher, ISBN, series, and original title.
- Elasticsearch `douban`: 904,763 documents. This is a derived search index. Import only missing IDs as low-confidence candidate records if needed.
- Elasticsearch `weixin`: 71,511 documents. This is a thin subset of MySQL `weixin`; do not migrate except for audit comparison.
- MySQL `jdbook` and `zybook`: useful historical competitor/store data. Move to `source_records` and optional `store_offers`, not core book identity in phase 1.

## Target Architecture

Use PostgreSQL as the only required data store.

- Structured relational tables hold books, editions, contributors, WeRead shelf items, notes, progress, matches, booklists, and purchases.
- `jsonb` keeps every raw source payload in `source_records`.
- `pg_trgm` and simple `to_tsvector` indexes replace the current Elasticsearch usage.
- `pgvector` is optional for phase 2 matching. The schema draft leaves it commented out.
- Cover/image comparison is done as a pipeline that writes evidence into `match_candidates`, not directly into the final match table.

Schema draft: `docs/legacy-migration/postgres_target_schema.sql`.

## Migration Phases

### Phase 0: Freeze and Export

- Keep the old app read-only.
- Generate a fresh audit with:

```bash
python3 tools/audit_legacy_data.py
```

- Export MySQL tables with consistent snapshot.
- Restore or open MongoDB with a compatible MongoDB 6/7 binary to inspect live collections. The current local `mongod` is 8.0.5 and refuses the existing FCV 6.0 data directory.
- Export Elasticsearch only after MySQL and Mongo exports are complete.

### Phase 1: Build PostgreSQL Foundation

- Create a new repo or clean package for the rebuild.
- Apply the initial PostgreSQL schema derived from `postgres_target_schema.sql`.
- Use `tools/migrate_legacy_sources.py` for first-pass ingestion into `source_records` only. It defaults to dry-run, reads legacy MySQL in a read-only transaction, reads Mongo from BSON, and refuses writes to the default PostgreSQL database unless explicitly allowed.
- Add checksum-based idempotency using `source_system`, `source_name`, `source_pk`, and `payload_hash`.
- Add import summary tables or logs before projecting normalized records.
- Follow `books/docs/safe_migration_runbook.md` for the current safe import sequence.

Success criteria:

- All MySQL priority tables are represented in `source_records`.
- Mongo BSON `local.book` imports all 490,229 documents.
- Re-running import produces no duplicate source records.

### Phase 2: Project Core Book Data

- Project Mongo Douban documents to `editions`, `contributors`, `edition_contributors`, `tags`, `edition_tags`, and `external_ids`.
- Project old SQL `books` only where it adds status fields or fills missing data.
- Project MySQL `weixin` to `weread_items`.
- Treat WeRead IDs as text, not integer, so `MP_WXS_*`, albums, and future non-book IDs can exist.

Success criteria:

- `editions` has one row per imported Douban ID where possible.
- `external_ids` has stable unique rows for `douban_id`, `isbn10`, `isbn13`, and `weread_book_id`.
- Search by title/author/publisher works from PostgreSQL without Elasticsearch.

### Phase 3: Preserve Curation and User Data

- Import `weixin_douban` into `book_matches`.
- Import booklists and booklist items.
- Import payment history.
- Import `author/meta_book/weixin_meta_book` as curated contributor/work evidence.

Success criteria:

- Count parity for `weixin_douban`, `weixin_booklist_book`, and `weixin_pay_history`.
- Every legacy match links either to an existing `weread_item` and Douban `edition`, or is marked unresolved with raw evidence.

### Phase 4: New WeRead Skill Sync

- Implement new sync against the local WeRead skill:
  - shelf sync
  - book info
  - progress
  - chapters
  - notebooks/bookmarks/reviews/underlines
  - reading data
- Store every response in `source_records` first.
- Then update projected tables.

Success criteria:

- Shelf count follows the skill rule: `books.length + albums.length + (mp present ? 1 : 0)`.
- Albums and `mp` are not dropped.
- Progress stores integer percent correctly; only `100` means completed.
- Reading time is stored in seconds.

### Phase 5: Matching Pipeline

- Generate candidates by deterministic rules first:
  - ISBN exact match
  - normalized title + author
  - title + translator + publisher
  - publication date proximity
- Add trigram/FTS search for broad candidates.
- Add embedding/image/OCR comparison for cover-heavy uncertain cases.
- Use a model to produce `model_score`, `model_reason`, and structured evidence.
- Only write final decisions to `book_matches` after threshold or review.

Success criteria:

- Legacy confirmed matches are preserved.
- New candidates are explainable.
- Low-confidence matches stay in `needs_review`.

## What Not To Carry Forward

- Do not keep MySQL + MongoDB + Elasticsearch as runtime architecture.
- Do not use ES as the canonical search system.
- Do not rebuild the 2025 `books/backend` and `books/frontend` half-refactor as the base.
- Do not model WeRead book IDs as numeric only.
- Do not let LLM/image comparison directly overwrite confirmed matches.
